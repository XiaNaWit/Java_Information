# 事务

## 1. 什么是数据库事务？

事务是一个不可分割的数据库操作序列，也是数据库并发控制的基本单位，其执行结果必须使数据库从一种一致性状态变到另一种一致性状态。事务是逻辑上的一组操作，要么都执行，要么都不执行。

例如转账：小明给小红转 1000 元，涉及"小明余额减 1000"和"小红余额加 1000"两个操作。事务保证这两个操作要么都成功，要么都失败。

## 2. 事务的 ACID 特性

| 特性 | 说明 | 实现手段 |
|:---|:---|:---|
| **原子性（Atomicity）** | 事务被视为不可分割的最小单元，所有操作要么全部成功，要么全部回滚 | Undo Log（回滚日志） |
| **一致性（Consistency）** | 事务执行前后数据都保持一致状态 | 原子性 + 隔离性 |
| **隔离性（Isolation）** | 一个事务的修改在提交前对其它事务不可见 | 锁机制 + MVCC |
| **持久性（Durability）** | 事务提交后修改永久保存 | Redo Log（重做日志） |

ACID 四个特性并非平级关系：
- 只有满足**一致性**，事务的执行结果才是正确的
- 无并发时，事务串行执行，满足隔离性。此时只需满足原子性，就能满足一致性
- 有并发时，事务需同时满足**原子性**和**隔离性**，才能满足一致性
- **持久化**是为了应对系统崩溃的情况

## 3. 事务隔离级别

SQL 标准定义了四种隔离级别，由低到高依次为：

### 未提交读（READ UNCOMMITTED）

事务中的修改，即使没有提交，对其它事务也是可见的。存在**脏读**问题。

### 提交读（READ COMMITTED）

一个事务只能读取已经提交的事务所做的修改。可以避免脏读，但可能出现**不可重复读**。

> 大部分数据库系统的默认隔离级别，如 Oracle、PostgreSQL、SQL Server。

### 可重复读（REPEATABLE READ）

保证在同一个事务中多次读取同一数据的结果一致。可以避免脏读和不可重复读，但可能出现**幻读**。

> MySQL InnoDB 存储引擎的默认隔离级别。

### 可串行化（SERIALIZABLE）

强制事务串行执行，互不干扰，可避免所有并发一致性问题，但性能最差。

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|:---|:---:|:---:|:---:|
| READ UNCOMMITTED | 可能 | 可能 | 可能 |
| READ COMMITTED | 不可能 | 可能 | 可能 |
| REPEATABLE READ | 不可能 | 不可能 | 可能 |
| SERIALIZABLE | 不可能 | 不可能 | 不可能 |

## 4. 并发一致性问题

### 丢失修改

一个事务的更新操作被另一个事务的更新操作覆盖。例如 T1 和 T2 同时对数据 A 进行修改，T1 先修改并提交，T2 随后修改覆盖了 T1 的修改。

### 读脏数据

当前事务可以读到其它事务**未提交**的数据。例如 T1 修改数据但未提交，T2 随后读取该数据。如果 T1 撤销修改，T2 读到的就是脏数据。

### 不可重复读

在一个事务内多次读取同一数据集合。在事务还未结束前，另一事务对该数据做了修改并提交，导致第一次事务两次读取的数据不一致。

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191207222102010.png"/> </div><br>

### 幻读

本质上也属于不可重复读，T1 读取某个范围的数据，T2 在这个范围内插入新的数据，T1 再次读取时发现比第一次多了行记录。

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191207222134306.png"/> </div><br>

### 对比总结

| 问题 | 原因 | 解决方法 |
|:---|:---|:---|
| 丢失修改 | 写操作互相覆盖 | 行锁 / 乐观锁 |
| 脏读 | 读到未提交数据 | 读已提交及以上隔离级别 |
| 不可重复读 | 其他事务修改了数据 | 可重复读及以上隔离级别 |
| 幻读 | 其他事务插入/删除了数据 | 串行化 / Gap Lock |
