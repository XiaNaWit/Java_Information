# 数据库锁

## 1. 为什么要加锁？

当多个用户并发地存取数据时，数据库中会产生多个事务同时存取同一数据的情况。若对并发操作不加控制，可能会读取和存储不正确的数据，破坏数据库的一致性。加锁的目的是保证多用户环境下数据库的完整性和一致性。

## 2. 锁的粒度

| 粒度 | 说明 | 特点 |
|:---|:---|:---|
| **行级锁** | 锁定单行记录 | 并发度高，加锁开销大，可能出现死锁 |
| **页级锁** | 锁定相邻的一组记录 | 介于行锁和表锁之间 |
| **表级锁** | 锁定整张表 | 并发度最低，加锁开销小，不会死锁 |

行锁冲突概率最低、并发度最高，但加锁慢、开销大；表锁正好相反；页锁是两者之间的折衷。

## 3. 锁的类型

### 共享锁（S锁 / 读锁）

多个事务可以同时加共享锁，彼此不冲突。加共享锁期间只能读，不能写。

| 兼容性 | S | X |
|:---:|:---:|:---:|
| S | ✓ | ✗ |
| X | ✗ | ✗ |

### 排他锁（X锁 / 写锁）

只有一个事务可以加排他锁，与其它所有锁互斥。通常用于 INSERT、UPDATE、DELETE 操作。

**规则**：
- 事务对数据 A 加了 X 锁，可读取和更新 A，期间其它事务不能对 A 加任何锁
- 事务对数据 A 加了 S 锁，可读取 A，但不能更新。期间其它事务可加 S 锁，不能加 X 锁

## 4. 意向锁

意向锁（Intention Locks）用于支持多粒度封锁。在同时存在行级锁和表级锁的情况下，事务 T 想对表 A 加 X 锁，需先检测是否有其它事务对表 A 或表 A 中的任意行加了锁。若没有意向锁，就需要逐行检测，非常耗时。

| 类型 | 含义 |
|:---|:---|
| **IS** | 事务**想要**在表中的某行加 S 锁 |
| **IX** | 事务**想要**在表中的某行加 X 锁 |

**兼容性表**：

| | X | IX | S | IS |
|:---:|:---:|:---:|:---:|:---:|
| X | ✗ | ✗ | ✗ | ✗ |
| IX | ✗ | ✓ | ✗ | ✓ |
| S | ✗ | ✗ | ✓ | ✓ |
| IS | ✗ | ✓ | ✓ | ✓ |

任意 IS/IX 锁之间都兼容，因为它们只表示"想要加锁"，还未实际加锁。

## 5. 乐观锁与悲观锁

| 类型 | 策略 | 实现方式 | 适用场景 |
|:---|:---|:---|:---|
| **悲观锁** | 假定会发生并发冲突，操作前先加锁 | 数据库的锁机制（SELECT ... FOR UPDATE） | 写多读少，冲突频繁 |
| **乐观锁** | 假定不会发生冲突，提交时检查 | 版本号机制 / CAS 算法 | 读多写少，冲突较少 |

## 6. 死锁

**死锁**是指两个或多个事务在同一资源上相互占用，并请求锁定对方的资源，从而导致恶性循环的现象。

**解决策略**：
1. 约定以相同顺序访问多个表
2. 在同一个事务中尽量一次锁定所需的所有资源
3. 对容易产生死锁的业务部分，可升级锁粒度（如表级锁）来减少死锁概率
4. 使用分布式事务锁或乐观锁

## 7. 三级封锁协议

通过约定何时加锁、何时释放锁，来解决并发一致性问题：

| 级别 | 规则 | 解决的并发问题 |
|:---|:---|:---|
| **一级** | 修改数据时必须加 X 锁，直到事务结束才释放 | 丢失修改 |
| **二级** | 一级 + 读取数据时加 S 锁，读完立即释放 | + 读脏数据 |
| **三级** | 一级 + 读取数据时加 S 锁，直到事务结束才释放 | + 不可重复读 |

## 8. 两段锁协议

加锁和解锁分为两个阶段进行。**可串行化调度**是指并发执行的事务结果与某个串行执行的事务结果相同。事务遵循两段锁协议是保证可串行化调度的**充分条件**（但不是必要条件）。

例如以下操作满足两段锁协议：
```
lock-x(A)...lock-s(B)...lock-s(C)...unlock(A)...unlock(C)...unlock(B)
```

大多数数据库系统会在需要时自动加锁（隐式锁定），所有锁在事务结束时统一释放。
