# SQL 优化

## 1. 大表数据查询优化

当单表数据量较大时，可采取以下优化策略：

- **索引优化**：合理创建索引，使用覆盖索引避免回表
- **缓存**：引入 Redis/Memcached 等缓存，减少数据库查询
- **读写分离**：主库负责写，从库负责读
- **垂直拆分**：按业务模块拆分表，分离使用频率低的字段
- **水平拆分**：按 Sharding Key 将数据分布到多张表或多个库

## 2. 超大分页优化

```sql
-- 低效写法（需要 load 大量无效行）
SELECT * FROM table WHERE age > 20 LIMIT 1000000, 10;

-- 优化：利用延迟关联或子查询（利用覆盖索引快速定位 id）
SELECT * FROM table WHERE id IN (
    SELECT id FROM table WHERE age > 20 LIMIT 1000000, 10
);
-- 或：
SELECT a.* FROM table a INNER JOIN (
    SELECT id FROM table WHERE age > 20 LIMIT 1000000, 10
) b ON a.id = b.id;
```

核心思路：先快速定位需要的 ID 段，再回表查询完整数据。

## 3. 查询优化

### 减少数据访问

- 使用 LIMIT 限制返回行数
- 指定列名代替 `SELECT *`
- 对重复查询使用缓存
- 利用 Explain 分析执行计划，避免全表扫描（type=ALL）

### 索引使用

- 对 WHERE、JOIN、ORDER BY 涉及的列建立索引
- 使用覆盖索引（Extra 出现 `Using index`）避免回表
- 避免索引失效：!=、类型不一致、函数包裹列、OR 连接等

### 关联查询优化

- 确保 ON/USING 子句中的列有索引
- 确保 GROUP BY 和 ORDER BY 只涉及同一个表的列，以便使用索引
- 小表驱动大表

## 4. 数据库结构优化

### 字段拆分

对于字段较多的表，可将使用频率低的字段分离成新表，避免查询大表时扫描不必要的数据。

### 增加中间表

对于需要频繁联合查询的场景，建立中间表存储关联结果，将联合查询改为单表查询。

### 合理冗余

范式化可以减少数据冗余，但查询需要 JOIN。适当的反范式化（冗余字段）可提高查询效率，但需注意数据一致性问题。

## 5. SQL 编写建议

### 避免 SELECT *

- 增加解析成本
- 无法使用覆盖索引
- 增加网络传输开销

### count 使用

| 写法 | 是否统计 NULL | 性能 |
|:---|:---:|:---:|
| `COUNT(*)` | 否 | 最快，InnoDB 做了优化 |
| `COUNT(1)` | 否 | 与 COUNT(*) 几乎相同 |
| `COUNT(col)` | 是（忽略 NULL） | 取决于列是否有索引 |

### 其他建议

- 使用 `UNION ALL` 代替 `UNION`（去重有额外开销）
- 减少 JOIN 表的数量
- 对 `GROUP BY` 列建索引
- 批量插入代替逐条插入
- 使用连接查询代替子查询
