# MySQL 锁

## 1. MyISAM 与 InnoDB 锁机制对比

| 存储引擎 | 默认锁粒度 | 说明 |
|:---|:---|:---|
| **MyISAM** | 表级锁 | 不支持事务，锁定整张表 |
| **InnoDB** | 行级锁 | 支持事务，也支持表级锁 |

> 锁的通用概念（粒度、类型、意向锁、乐观/悲观锁、死锁、封锁协议等）参见 [锁.md](../理论/锁.md)。

## 2. InnoDB 行锁实现

InnoDB 行锁是通过给**索引项**加锁来实现的。

```sql
-- id 有索引 → 行锁
SELECT * FROM tab_with_index WHERE id = 1 FOR UPDATE;

-- id 无索引 → 表锁（退化为表级锁）
SELECT * FROM tab_without_index WHERE id = 1 FOR UPDATE;
```

InnoDB 行锁的三种实现：

| 锁类型 | 说明 |
|:---|:---|
| **Record Lock（记录锁）** | 锁定单个行记录 |
| **Gap Lock（间隙锁）** | 锁定索引之间的间隙，不包括记录本身 |
| **Next-Key Lock（临键锁）** | Record Lock + Gap Lock，锁定一个前开后闭区间 |

## 3. 隔离级别与锁的关系

| 隔离级别 | 读操作加锁行为 |
|:---|:---|
| READ UNCOMMITTED | 读操作不加共享锁 |
| READ COMMITTED | 读操作加共享锁，语句执行完立即释放 |
| REPEATABLE READ | 读操作加共享锁，事务提交后才释放 |
| SERIALIZABLE | 锁定整个范围的键，直到事务完成 |

## 4. MySQL 锁优化建议

- 尽量使用较低的隔离级别
- 合理设计索引，使加锁更加精确，减少锁冲突
- 选择合理的事务大小，一次性请求足够级别的锁
- 不同程序访问同一组表时，约定相同的访问顺序，减少死锁
- 尽量使用等值条件访问数据，避免间隙锁对并发插入的影响
- 不要申请超过实际需要的锁级别
- 查询时非必要不用加锁（利用 MVCC 的快照读）
- 对于特定事务，可使用表锁来提高处理速度或减少死锁概率
